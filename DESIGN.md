## Problem Statement

Applied to [busonlineticket.co.th](https://www.busonlineticket.co.th/th/) — didn't get the job. Decided to build an open-source version of what they do: a booking engine that any developer can use to build their own transport booking system.

The gap is real: no open-source booking engine framework for transport exists. Agency devs in Thailand rebuild the same booking logic (search, seat map, reservation lock, checkout) for every operator client. `busonlineticket.co.th` proves the market exists — people book bus tickets online. But there's no reusable, open-source foundation.

## Demand Evidence

- Inferred from competitor existence: busonlineticket.co.th is a functioning, presumably profitable business → market exists
- PRD identifies two user groups: developers who rebuild booking logic per-project, and operators who can't afford proprietary systems
- Not yet validated with real users — this is pre-product, pre-revenue
- **Honest assessment:** Demand is inferred, not proven. The first validation step is shipping the package and seeing if any agency actually uses it.

## Status Quo

Current workflow when a Thai agency builds a booking system for a transport operator:

1. Start from scratch (Laravel/Django/Express scaffolding)
2. Build route search, seat map, reservation logic, checkout flow manually
3. Repeat for next client with minor variations
4. No reusable component across projects

No existing open-source library solves this. Two npm packages exist for seat maps (`@jaganax/seat-chart-reactjs`, `@alisaitteke/seatmap-canvas`) but neither handles booking logic, reservation locks, or provider abstraction.

## Target User & Narrowest Wedge

**Primary user:** A dev at a Thai agency who builds booking websites for bus/van operators. They need a reusable core engine they can wrap with their own UI and API.

**Narrowest wedge:** `@bef/core` — a single npm package, pure TypeScript (ESM-only, dual-publish not needed for Node 18+ target), zero framework dependencies. Contains:

- `TripProvider` interface (the adapter contract)
- `BookingService` (create, confirm, cancel — event-sourced, see Event Sourcing Model below)
- `SeatLockService` (interface with in-memory + pluggable Redis adapter, see Seat Lock Specification below)
- `MockProvider` (demo data for dev and test, see MockProvider Fixtures below)
- Full test coverage (see Test Scope below)

**Out of scope for v1:**
- No NestJS API layer (`@bef/api`)
- No Next.js frontend (`@bef/web`, `@bef/admin`)
- No Docker Compose
- No real provider implementations (BusProvider, TrainProvider, BoatProvider)
- No WebSocket seat availability
- No OpenAPI spec (deferred to post-v1 — v1 documentation via TSDoc/TypeDoc)
- No CI/CD automation (v1 publishes manually; CI added when velocity demands it)

## Core Types & Public API

### Domain Types

```typescript
// ---- Seats ----

// Physical status from the provider (persistent)
type SeatPhysicalStatus = 'available' | 'unavailable'; // e.g. seat exists, seat broken, seat not in service

// Runtime status after overlay (transient, derived from locks + bookings)
type SeatRuntimeStatus = 'available' | 'locked' | 'booked' | 'unavailable';

interface Seat {
  id: string;              // e.g. "1A", "12B"
  row: number;
  column: string;
  deck: 'upper' | 'lower'; // bus-specific; extensible via metadata on Trip
  status: SeatPhysicalStatus; // physical inventory only — use BookingService.getLiveSeatMap() for runtime
  metadata?: Record<string, unknown>;
}

// ---- Trips ----

interface Trip {
  id: string;
  origin: string;
  destination: string;
  departureTime: Date;
  arrivalTime: Date;
  operator: string;
  vehicleType: 'bus' | 'van' | 'train' | 'boat' | 'ferry';
  seats: Seat[];
  metadata?: Record<string, unknown>; // mode-specific fields
}

// ---- Events ----

type BookingEvent =
  | { type: 'BookingCreated'; bookingId: string; tripId: string; seatIds: string[]; userId: string; timestamp: Date }
  | { type: 'BookingConfirmed'; bookingId: string; timestamp: Date }
  | { type: 'BookingCancelled'; bookingId: string; reason?: string; timestamp: Date }
  | { type: 'SeatLocked'; seatId: string; bookingId: string; expiresAt: Date; timestamp: Date }
  | { type: 'SeatReleased'; seatId: string; bookingId: string; timestamp: Date };

// ---- Booking ----

type BookingStatus = 'pending' | 'confirmed' | 'cancelled' | 'expired';

interface Booking {
  id: string;
  tripId: string;
  seatIds: string[];
  userId: string;
  status: BookingStatus;
  events: BookingEvent[];
  createdAt: Date;
  updatedAt: Date;
}
```

### TripProvider Interface

```typescript
interface TripSearchQuery {
  origin?: string;
  destination?: string;
  departureDate?: Date;
  vehicleType?: Trip['vehicleType'];
}

interface TripSummary {
  id: string;
  origin: string;
  destination: string;
  departureTime: Date;
  arrivalTime: Date;
  operator: string;
  vehicleType: Trip['vehicleType'];
  availableSeatCount: number;
}

interface TripSearchResult {
  trips: TripSummary[];
  total: number;
}

interface TripProvider {
  searchTrips(query: TripSearchQuery): Promise<TripSearchResult>; // lightweight — no seat arrays
  getTrip(tripId: string): Promise<Trip | null>;                  // full detail with seats
  getSeatMap(tripId: string): Promise<Seat[]>; // returns physical inventory only — status is 'available' | 'unavailable'
}
```

`TripProvider.getSeatMap()` returns physical inventory from the operator — `status` reflects whether the seat physically exists and is in service, NOT runtime lock state. Runtime state (locked/booked) is overlaid by `BookingService.getLiveSeatMap(tripId)` which merges 3 sources with this priority:

1. **Bookings win** (confirmed bookings → `booked`; pending bookings whose lock still held → `locked`)
2. **Locks win over provider** (locked seats not yet booked → `locked`)
3. **Provider is baseline** (everything else → `available` | `unavailable`)

No cross-source assertions — graceful overlay handles transient states (e.g., lock expired but booking still pending).

### BookingService Interface

```typescript
interface BookingService {
  new (tripProvider: TripProvider, seatLockService: SeatLockService): BookingService;
  createBooking(tripId: string, seatIds: string[], userId: string): Promise<Booking>;
  confirmBooking(bookingId: string): Promise<Booking>;
  cancelBooking(bookingId: string, reason?: string): Promise<Booking>;
  getBooking(bookingId: string): Promise<Booking | null>;
  getLiveSeatMap(tripId: string): Promise<Seat[]>; // TripProvider.getSeatMap() +
                                                     // lock state overlay + booking records
  releaseExpiredBookings(): Promise<Booking[]>;
}
```

Constructor injection: `BookingService` receives `TripProvider` and `SeatLockService` at construction time. No DI container required — plain TypeScript. Both mockable in tests.

`releaseExpiredBookings()` scans all `pending` bookings, checks each seat's lock via `SeatLockService.isLocked()`, and auto-cancels any with expired locks. Consumer owns scheduling (setInterval, cron, etc.) — `BookingService` remains passive/pure.

**Multi-seat booking is all-or-nothing atomic.** `createBooking` locks all `seatIds` in order. If any `acquireLock` returns false, all previously-acquired locks are released (inverse order), and the method throws `SEAT_UNAVAILABLE` with the failing seat ID in the message. No partial bookings — consumer retries.

### SeatLockService Interface

```typescript
interface SeatLockService {
  acquireLock(seatId: string, bookingId: string, ttlMs: number): Promise<boolean>;
  releaseLock(seatId: string, bookingId: string): Promise<void>;
  renewLock(seatId: string, bookingId: string, ttlMs: number): Promise<boolean>;
  isLocked(seatId: string): Promise<boolean>;
  getLockOwner(seatId: string): Promise<string | null>;
}
```

### Error Model

```typescript
class BookingError extends Error {
  constructor(
    message: string,
    public code: 'SEAT_UNAVAILABLE' | 'LOCK_EXPIRED' | 'BOOKING_NOT_FOUND'
      | 'INVALID_STATE_TRANSITION' | 'CONCURRENT_MODIFICATION',
  ) {
    super(message);
  }
}
```

## Event Sourcing Model (v1)

v1 uses an in-memory append-only event log within each `Booking` aggregate. No persistence, no snapshots, no replay — deliberately simplest correct thing.

- **Events appended to `Booking.events[]`** during `createBooking`, `confirmBooking`, `cancelBooking`.
- **State derived via pure-function reducer**: `reduceBooking(events: BookingEvent[]): BookingStatus`.
- **All mutations are atomic**: each method reads current state, validates transition, appends event, returns new state.
- **No side effects in reducers** — I/O (lock acquire/release) happens in service layer, not event handlers.
- **v1 storage**: Private `Map<string, Booking>` inside `BookingService`. No persistence, no `BookingStore` interface — simplest correct thing. The event log in `Booking.events[]` is the persistence seed for v2.
- **Optional `EventStore` interface** is a v2 concern for persistence (PostgreSQL append-only table or similar).

### State Machine

```
                  createBooking()
  (none) ───────────────────────────> pending
                                       │
                          confirmBooking() │  cancelBooking() / lock expiry
                                       │         │
                                       v         v
                                   confirmed  cancelled
```

Invalid transitions (e.g. confirming an already-cancelled booking) throw `BookingError` with code `INVALID_STATE_TRANSITION`.

## Seat Lock Specification

### Semantics

- **Pessimistic locking**: `acquireLock` uses atomic check-and-set. A seat can only be locked by one `bookingId` at a time.
- **Lock key**: `seat:{seatId}:lock` → value is `bookingId`.
- **Default TTL**: 10 minutes (600,000 ms). Configurable per call.
- **On expiry**: lock is released atomically by the store. Booking transitions to `cancelled` if consumer calls `cancelBooking` before `confirmBooking`. Lock expiry does NOT auto-cancel the booking — the consumer detects expiration on next `confirmBooking` or `renewLock` call and handles it.
- **Atomicity guarantee**: In-memory impl uses `Map` with synchronous access (suitable for single-process testing). Redis impl uses `SET seat:{seatId}:lock {bookingId} NX PX {ttlMs}` with Lua script for release (check-then-delete in one round trip).
- **Lock renewal**: Call `renewLock` to extend TTL during long checkout flows.

### v1 In-Memory Implementation

```typescript
class InMemorySeatLockService implements SeatLockService { /* ... */ }
```

Uses a `Map<string, { bookingId: string; expiresAt: number }>` with `setTimeout` cleanup. Suitable for development, testing, and single-process deployments. Not suitable for multi-process production (no cross-process atomicity).

### Production (Separate Package, Post-v1)

`@bef/seat-lock-redis` — implements `SeatLockService` using ioredis. Ships as a separate npm package so `@bef/core` remains zero runtime dependencies. Consumers opt in: `npm install @bef/core @bef/seat-lock-redis`.

## Test Scope

**"Core" defined as:** domain types, event reducers, `BookingService` state machine, `SeatLockService` logic, `MockProvider` fixtures. **Excluded from core coverage target:** I/O adapters, transport layer, framework integrations.

- Domain types: type-level tests (TypeScript compilation verifies type correctness)
- Reducer: pure function tests — given event sequence X, expect state Y
- BookingService: state machine tests — valid transitions pass, invalid transitions throw `BookingError`
- SeatLockService: in-memory implementation — acquire, release, renew, expiry, double-acquire rejection
- MockProvider: each fixture returns correct shape, edge case fixtures exercise boundary conditions

## Concurrency Test Plan

The hardest correctness guarantee: "zero double bookings under concurrent seat reservation."

1. **Test harness**: Single process, `InMemorySeatLockService`, N concurrent `Promise.all([...createBooking(...)])` calls for the same seat.
2. **Scenario 1 — N-way race on last seat**: 10 concurrent attempts to book the single remaining seat. Exactly 1 succeeds; 9 get `SEAT_UNAVAILABLE`.
3. **Scenario 2 — Lock expiry mid-checkout**: Lock acquired, TTL set to 100ms, sleep 150ms, attempt confirm → `LOCK_EXPIRED`.
4. **Scenario 3 — Confirm after cancel**: Booking cancelled, then confirm attempted → `INVALID_STATE_TRANSITION`.
5. **Scenario 4 — Idempotent confirm**: Same booking confirmed twice → second call returns existing confirmed booking (no error).
6. **Target**: These scenarios run as part of `vitest` suite, no external Redis required. Redis concurrency tests are post-v1 when `@bef/seat-lock-redis` is built (using testcontainers).

## Constraints

- Solo developer
- No existing users or customers
- MIT license, open-source
- Target: npm package that works with any Node.js framework
- Pre-product stage — focus on shipping, not perfection
- TypeScript `strict` mode, zero runtime production dependencies
- ESM-only (`"type": "module"`, `"exports"` map in package.json). CJS not needed for Node 18+ target audience.
- Accepting the zero-deps tradeoff: manual type guards instead of zod, manual Result patterns, simple `crypto.randomUUID()` for IDs. This is explicit — we trade more code for zero install friction.

## Premises

1. **Agency devs will use a reusable npm package** — if it solves the genuinely hard parts (seat map state machine, lock timer, event sourcing), they'd rather import than rebuild
2. **Pure TypeScript, zero deps = maximum adoption** — works with Express, NestJS, Fastify, Next.js API routes, whatever they already use
3. **MockProvider is enough for v1** — proves the interface works without building real provider integrations
4. **Open-source MIT + npm publish = distribution** — GitHub stars + npm downloads = validation signal
5. **v1 only needs bus mode** — the provider interface proves extensibility; community can add train/boat/van later. MockProvider includes one van fixture to demonstrate mode independence.

## Recommended Approach

**Approach A: @bef/core npm package first.**

Rationale: Start with the irreducible core. A framework-agnostic, zero-dependency npm package that solves the genuinely hard problems (booking state machine, seat lock, provider abstraction) is deliverable in weeks, not months. It validates the interface design before committing to a full-stack implementation. If nobody uses the npm package, the full Docker Compose approach wouldn't have worked either — but you learn that in 3 weeks instead of 6 months.

The package is also the foundation everything else builds on. Approaches B and C both need `@bef/core` to exist. Ship it first, see if anyone cares, then decide whether to build the API layer, frontend, and Docker setup.

## Design Decisions Log (grill-me session, 2026-06-05)

1. **Constructor injection** — `BookingServiceImpl` takes `TripProvider` and `SeatLockService` at construction. No DI container. Mock both in tests.
2. **In-memory booking storage** — private `Map<string, Booking>` inside `BookingServiceImpl`. No `BookingStore` interface in v1. `Booking.events[]` is the persistence seed for v2.
3. **`releaseExpiredBookings()`** — passive sweep method. Consumer owns scheduling (setInterval/cron). No internal timers in `BookingServiceImpl`.
4. **All-or-nothing multi-seat booking** — lock all `seatIds` first. Any failure → release all, throw `SEAT_UNAVAILABLE`. No partial bookings.
5. **Seat status separation** — `SeatPhysicalStatus` ('available' | 'unavailable') from provider. `SeatRuntimeStatus` from `getLiveSeatMap()` overlay (bookings > locks > provider). `TripProvider` never returns lock/booking state.
6. **Search returns summaries** — `searchTrips()` returns `TripSummary[]` (no seat arrays). `getTrip()` for full detail. `getLiveSeatMap()` for runtime seat state.
7. **MockProvider zero-config** — 6 fixtures loaded at construction, discoverable via `searchTrips`, loadable by ID via `getTrip`.
8. **`getLiveSeatMap` overlay priority** — bookings win > locks win > provider baseline. Graceful overlay, no cross-source assertions.
9. **Interface/Implementation separation** — `BookingService` interface + `BookingServiceImpl` class. `SeatLockService` interface + `InMemorySeatLockService` class. Consumer types against interfaces, constructs classes. `reduceBooking`: pure function export.
10. **Flat source layout** — 8 source files, 5 test files, no subdirectories. `index.ts` barrel.
